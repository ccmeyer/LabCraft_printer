# Firmware — Agent Instructions (STM32 / CubeIDE)

## Mission
Safely modify and validate the STM32 firmware. This firmware controls real hardware.
Changes must be minimal, reviewable, and verified via automated checks.

This repo supports three validation lanes:
1) **Host unit tests** (CppUTest) — fast, runs on the PC (`tests_host`)
2) **Headless CubeIDE build** — compiles/links real firmware and updates the `.bin` artifact
3) **Hardware-in-the-loop (HIL) self-test** — flashes the MCU via the Pi and runs SAFE self-test, producing a JSON report

## Firmware layout
- CubeIDE project root is `firmware/` (contains `.project`, `.cproject`, and usually `.ioc`)
- Source:
  - `Core/Inc` and `Core/Src` (or `Inc/` + `Src/`, depending on the project)
- Host tests:
  - `tests_host/` (CMake + CppUTest runner)
  - `third_party/cpputest/` (CppUTest framework)
- Scripts:
  - `scripts/run_fw_unit_tests.ps1`
  - `scripts/build_firmware_headless.ps1`
  - `scripts/run_fw_checks.ps1`
  - `scripts/run_fw_hil_windows.ps1`  ← deploys to Pi, flashes, runs self-test, downloads report
- HIL runner (on Pi):
  - `firmware/hil/flash_and_test.sh`
- Self-test host runner:
  - `tools/run_selftest.py` (runs on Pi, talks to MCU, writes JSON report)
- Artifacts:
  - `firmware/artifacts/` contains copied build outputs (e.g., `LabCraft_firmware.bin`)
- Reports:
  - `hil_reports/` contains self-test JSON reports and logs (local + on Pi)

## Non-negotiable safety rules
- **Do not change protocol formats** (message IDs, payload layouts, parsing rules) unless explicitly requested.
- Avoid changes that could cause unsafe motion/pressure behavior.
- Keep diffs small. Prefer local changes to one module at a time.
- SAFE self-test must remain **non-destructive** (no motion, no valve actuation, no pressure changes) unless explicitly implementing FULL profile.

## Generated code rule (CubeMX / CubeIDE)
Many files are generated by CubeMX.
**Only modify code inside `/* USER CODE BEGIN */ ... /* USER CODE END */` blocks** in generated files.
Do not edit outside those blocks unless the file is clearly hand-written and not generated.
Do not edit `.ioc` unless explicitly requested.

If you are unsure whether a file is generated:
- search for `USER CODE BEGIN` markers
- assume it is generated and restrict edits to those regions

## Validation required after any firmware edit

### Required before commit: full HIL gate (preferred)
Run the complete local→Pi→flash→selftest loop from Windows:
`powershell -ExecutionPolicy Bypass -File firmware/scripts/run_fw_hil_windows.ps1 -PiHost labcraft.local`

This command should:
1) run host unit tests + headless build (via `run_fw_checks.ps1`)
2) upload `firmware/artifacts/LabCraft_firmware.bin` and `tools/run_selftest.py` to the Pi
3) run `firmware/hil/flash_and_test.sh` on the Pi
4) download the JSON self-test report to local `hil_reports/`
5) exit **0** on PASS, non-zero on FAIL

A change is not “done” until this passes (or you explicitly explain why HIL cannot run and provide an alternative verification plan).

### Local lane (required at minimum, before pushing code)
`powershell -ExecutionPolicy Bypass -File firmware/scripts/run_fw_checks.ps1 -Config Debug`

If `run_fw_checks.ps1` is unavailable, run both:
- `powershell -ExecutionPolicy Bypass -File firmware/scripts/run_fw_unit_tests.ps1 -Config Debug`
- `powershell -ExecutionPolicy Bypass -File firmware/scripts/build_firmware_headless.ps1`

### Fast lane (optional while iterating)
`powershell -ExecutionPolicy Bypass -File firmware/scripts/run_fw_unit_tests.ps1 -Config Debug`

## How to use self-test results to guide changes

### Where results live
After a HIL run, a JSON report is written locally under:
- `hil_reports/selftest_<timestamp>.json`

The report schema includes:
- `summary.total`, `summary.passed`, `summary.failed`
- `aborted` (true means the run did not complete normally)
- `results[]` list of per-test entries:
  - `test_id` (stable numeric ID)
  - `name` (stable string)
  - `pass` (boolean)
  - `metrics` (key/value map; strings/numbers)

### How to interpret failures
- If `aborted=true` or the report is missing: treat as infrastructure/connection issue first (port, flash, protocol handshake).
- If `summary.failed > 0`: inspect the failing `results[]` entries and their `metrics`:
  - Use `test_id`/`name` to locate the firmware-side test implementation (self-test emitter) and the host parser (`tools/run_selftest.py`).
  - Fix should be targeted: either firmware behavior, test expectation, or reporting/encoding.
- If failures implicate protocol/framing/CRC, prioritize:
  - `CommCodec.*` + protocol tests in `tests_host`
  - ensure no protocol-breaking change was introduced

### Expected behavior
- SAFE self-test should be deterministic and complete quickly (< ~30s).
- Adding new SAFE tests is encouraged, but they must remain non-destructive unless implementing FULL profile explicitly.

## What to unit test (host tests)
Host tests should target **pure logic** that does not require STM32 peripherals:
- protocol encode/decode (golden vectors)
- framing/parsing
- ring buffers / queues
- CRC/checksum utilities
- state machines (homing/printing sequencing) when implemented without HAL calls
- math utilities (conversions, bounds checks, PID helper math if isolated)

Host tests should NOT attempt to compile:
- HAL peripheral init
- ISR code directly
- FreeRTOS task startup logic
- device drivers that require real hardware registers

### Stubbing policy
When a host-tested module includes HAL/FreeRTOS headers:
- add minimal stub headers under `tests_host/stubs/` matching the include filenames
- define only what is required to compile the module under test
- do not implement behavior unless a test needs it

## Preferred patterns for firmware changes
- Keep HAL/RTOS at the edges; move logic into small modules that can be compiled in `tests_host/`.
- If a function currently mixes I/O with logic, split into:
  - a pure function (unit tested)
  - a thin wrapper that calls HAL/RTOS
- When adding HIL tests, keep firmware-side self-test code small and table-driven:
  - stable `test_id` and `name`
  - structured metrics
  - explicit timeouts and abort handling

## Headless build notes
- Headless builds should run in an isolated temp workspace (handled by the script).
- The script should copy the newest `.bin` into `firmware/artifacts/` (as configured).
- Do not assume STM32CubeIDE is open on the same workspace during headless builds.

## Commit discipline
- One firmware milestone per commit (small diffs).
- Commit messages should be specific, e.g.:
  - `firmware: add SAFE selftest pressure idle check`
  - `firmware: add host tests for CommCodec parser recovery`
  - `build: improve HIL runner robustness`

Do not commit transient build outputs:
- `tests_host/build/`
- workspace metadata (`.metadata/`)
- temporary logs

Artifacts in `firmware/artifacts/` are only committed if the repo policy explicitly says to version `.bin` files.

## Definition of done (firmware task)
- Local checks pass:
  - `run_fw_checks.ps1 -Config Debug`
- HIL gate passes (preferred):
  - `run_fw_hil_windows.ps1 -PiHost labcraft.local`
- Update `firmware/docs/repo_map.md` if the change affects module ownership, entrypoints, or protocol handling.
- Summary includes:
  - what changed and why
  - what modules were touched
  - how it was validated (commands + results)
  - self-test report path and brief PASS/FAIL summary
  - risk/edge cases
  - rollback steps
